## 一、项目概况

网页版中国象棋对弈平台，2026 年 7 月 11 日从零开发并上线。

- **功能**：人机对战（三档难度）、在线人人对战（快速匹配 / 房号邀请）、5/10/15 分钟 + 每步加 3 秒计时、断线重连、认输/求和/再来一局
- **代码仓库**：GitHub `web-chinese-chess`（本地目录 `C:\Users\cx\Claude\Projects\web_chinese_chess`）
- **线上部署**：Render 免费档 Web Service（Region: Singapore），绑定 GitHub 仓库，**push 到 main 分支即自动重新部署**
- **访问统计**：Microsoft Clarity（项目 ID `xkpuw1esyh`，后台 clarity.microsoft.com）+ 自建 `/stats` 实时接口

## 二、技术架构

**技术选型原则：越简单越好。** 无前端框架、无构建步骤、无数据库，全部状态在服务器内存中。一个 Node 进程同时提供网页和对战服务。

```
浏览器 ──HTTP──> Express 静态文件（public/ 与 shared/）
浏览器 <─WebSocket─> 对战服务（同端口 /ws 路径）
```

| 文件 | 职责 |
|---|---|
| `shared/xiangqi.js` | 规则引擎，**前后端共用同一份代码**：走法生成、将军检测、绝杀/困毙判定、中文记谱 |
| `server/server.js` | 全部后端：静态托管、房间管理、快速匹配队列、走法校验、权威计时、断线重连、`/stats` |
| `public/js/main.js` | 前端总控制器：页面切换、人机/联机对局流程、时钟显示、按钮事件 |
| `public/js/board.js` | Canvas 棋盘渲染与点击换算（黑方视角自动翻转） |
| `public/js/ai-worker.js` | AI 引擎，跑在 Web Worker 里不卡界面：迭代加深 α-β 搜索 + 吃子静态搜索 + 子力位置表 |
| `public/js/net.js` | WebSocket 客户端：自动重连（指数退避）、会话恢复（sessionStorage 存房号+令牌）、离线消息排队 |

### 关键设计决策

1. **规则引擎前后端共用**——服务器用它校验联机走法（防作弊），前端用它做交互提示和 AI，一份代码不会出现两边规则不一致。
2. **联机计时以服务器为准**——每步走完服务器结算用时并加 3 秒，把最新时钟广播给双方；客户端只做显示插值。作弊者改不了自己的时间。
3. **人机模式全部在浏览器本地运行**——AI 在 Web Worker 里搜索，不占服务器资源；免费服务器只承担联机对战。
4. **断线重连**——服务器给每个玩家发 UUID 令牌，浏览器存在 sessionStorage；重连/刷新后凭令牌拿回完整对局状态，60 秒宽限期内不判负。
5. **快速匹配 = 隐藏的好友房**——每个时间档一个等待位，配对后完全复用好友房逻辑，专属代码仅约 20 行。

### AI 难度参数（main.js 里的 `AI_LEVELS`）

| 难度 | 思考时间 | 深度上限 | 随机扰动 |
|---|---|---|---|
| 简单 | 250ms | 2 层 | 大（会走出失误） |
| 中等 | 900ms | 10 层 | 小 |
| 困难 | 2600ms | 24 层 | 无（实测约 4-5 层 + 吃子延伸） |

## 三、开发与上线过程（含踩过的坑）

按时间顺序，**坑都值得记住**：

1. **规则引擎先行 + perft 验证**。走法生成用国际公认的象棋 perft 基准数验证：初始局面 1~4 层节点数 44 / 1920 / 79666 / 3290240 全部吻合，从根上保证规则正确。
2. **坑 ①：CSS 覆盖了 `hidden` 属性**。`.overlay { display:flex }` 优先级高于 HTML `hidden` 属性的 `display:none`，导致求和弹窗常驻显示。修复：加 `.overlay[hidden] { display:none; }`。**教训：给有 display 样式的元素用 hidden 属性时必须补这条规则。**
3. **坑 ②：隐藏状态下初始化 Canvas**。棋盘在 `display:none` 的页面里初始化时 `clientWidth` 为 0，格子尺寸算错，点击坐标全部错位。修复:进入对局页时同步调用 `resize()`，且点击时检测尺寸过期自动重算。
4. **上线：GitHub + Render**。过程中遇到 Start Command 误填 `node start`（正确是 `npm start`）导致部署起不来，报 `Cannot find module .../start`。
5. **坑 ③（最重要）：快速匹配"双方永远在等待"**。本地怎么测都正常，线上必现。根因：排队时 WebSocket 一旦断开（手机锁屏、切后台、代理断开空闲连接），服务器把玩家移出队列，客户端自动重连后**没有重新排队**，界面却还显示"匹配中"。修复：客户端记住匹配意图，重连后自动重新入队。**教训：本地网络永远稳定，一切"断线后怎么办"的逻辑必须刻意模拟测试。**
6. **在线统计**：自建访客心跳（页面每 30 秒 POST `/presence`，访客 ID 存 localStorage 去重）+ Clarity 长期报表。

### 查看线上状态

| 想知道什么 | 去哪看 |
|---|---|
| 服务是否活着 | Render 控制台是否绿色 Live；或访问 `你的网址/healthz` |
| 当前在线/对局数 | 访问 `你的网址/stats`（JSON） |
| 谁在排队、谁掉线了 | Render 控制台 → Logs（服务器有详细日志） |
| 每天多少人来玩、操作录像 | clarity.microsoft.com 后台 |

### 出了问题怎么回滚

```powershell
git log --oneline        # 找到上一个正常版本的编号（前 7 位）
git revert 出问题的提交编号   # 生成一个"撤销"提交，不破坏历史
git push                 # 自动重新部署为回滚后的版本
```

### 故障速查

| 症状 | 大概率原因 | 处理 |
|---|---|---|
| 网页打不开，转圈半分钟后恢复 | 免费档休眠唤醒（15 分钟无访问即休眠） | 正常现象；要消除需升级付费档 |
| 匹配不到人 | 看 Render Logs 是否两人都进了队列 | 日志里 `quick match: waiting/paired` 一眼可辨 |
| 部署失败 | 看 Render 部署日志报错行 | 常见：package.json 语法错误、Start Command 被改动 |
| 对局中频繁掉线 | 免费档实例重启（内存对局会丢失） | 观察 Logs 重启记录；频繁则考虑付费档 |

## 五、已知局限（做决定时要记得）

1. **所有对局状态在内存里**——服务器重启（每次部署、免费档休眠唤醒）会丢掉进行中的对局和 `/stats` 累计数。要持久化需引入数据库（如 SQLite/Postgres），属于较大改造。
2. **免费档休眠**——玩家少时体验为"第一个人要等半分钟"。
3. **重复局面判和是简化规则**——正式象棋规则里"长将判负"，目前实现为任何三次重复即判和，极少数对局结果与正规裁决不同。
4. **无账号系统**——没有积分、排行榜、按棋力匹配；访客统计基于浏览器 localStorage，换浏览器算新访客。
